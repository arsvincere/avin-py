== GUI Architecture

GUI module is built as a Qt Widgets desktop application.

The main goal of GUI architecture is to avoid spreading application state across widgets and to avoid direct widget-to-widget synchronization through a web of Qt signals.

AVIN GUI uses a hybrid architecture:

* Qt-native UI shell.
* MVU/Redux-style application state management.
* Qt Model/View for tables, trees and lists.
* QGraphicsScene/QGraphicsView for custom market visualizations.
* Service/domain/data layers remain independent from GUI.

=== Main principles

GUI must not own trading domain logic.

Domain objects such as `Tick`, `Level`, `Ladder`, `Cluster`, `Footprint`, `Iid` belong to `domain`, `data`, `service` and `analyse` layers.

GUI is responsible for:

* displaying application state;
* sending user actions;
* rendering market data;
* managing windows, docks, menus, toolbars and scene interaction.

GUI is not responsible for:

* storing market data as source of truth;
* building footprints directly inside widgets;
* loading/parsing files directly inside visual components;
* synchronizing widgets with each other directly.

=== UI shell

The main GUI shell is based on:

* `QApplication`
* `QMainWindow`
* `QDockWidget`
* menus
* toolbars
* status bar
* central widgets / dock widgets

Dock widgets are the preferred way to build the AVIN interface.

This allows the user to customize the workspace layout and later makes it possible to save and restore workspace state.

Typical future dock widgets:

* instrument dock;
* footprint dock;
* chart dock;
* state/debug dock;
* cluster inspector;
* level inspector;
* trades/tape dock;
* analyse dock;
* log dock.

Every dock widget must have stable `objectName`, because Qt uses it for saving and restoring dock layout.

=== Application state

GUI has a single application state object:

## [source,python]

## AppState

`AppState` is the single source of truth for global GUI state.

It may contain:

* current instrument;
* current data source;
* current timeframe;
* loaded ticks metadata;
* current footprint metadata;
* selected cluster;
* selected price level;
* active chart;
* opened panels/charts;
* runtime status;
* last action;
* error state.

`AppState` stores data only.
It does not load files, build footprints, emit Qt signals, draw widgets or call GUI code.

Prefer immutable state updates:

## [source,python]

## new_state = replace(old_state, current_iid=iid)

This makes state transitions explicit and easier to debug.

=== Actions

User events and system events are represented as actions.

Examples:

## [source,python]

SelectIid
SelectTimeframe
LoadTicks
BuildFootprint
SelectCluster
SelectPrice
ClearSelection
AppStarted
----------

Widgets do not change application state directly.

Instead, widgets dispatch actions:

## [source,python]

## store.dispatch(SelectIid(iid))

Actions describe what happened.
They do not decide how the whole application should change.

=== Store

`Store` owns the current `AppState`.

Responsibilities of `Store`:

* keep current application state;
* accept actions;
* create a new `AppState`;
* emit state changes to GUI subscribers;
* provide one official path for changing global GUI state.

Conceptual flow:

## [source,text]

Widget event
-> Action
-> Store.dispatch(action)
-> new AppState
-> state_changed
-> widgets render updated state
-------------------------------

Widgets must not synchronize other widgets directly.

Bad:

## [source,text]

InstrumentDock -> ChartDock
InstrumentDock -> InspectorDock
ChartDock -> StateDock
----------------------

Good:

## [source,text]

InstrumentDock -> Store -> AppState -> ChartDock
-> InspectorDock
-> StateDock
------------

Qt signals are still used, but mainly as technical delivery mechanism:

* widget event -> dispatch action;
* store state_changed -> widgets render state.

Signals must not become hidden business logic.

=== Widgets

Widgets should be mostly dumb.

Widget responsibilities:

* render state or view model;
* collect user input;
* dispatch actions to `Store`.

Widget should not:

* own global application state;
* know about other widgets;
* directly call other widgets;
* store current instrument as source of truth;
* build footprints;
* load large datasets;
* contain domain logic.

Local UI-only state may stay inside widgets.

Examples of acceptable local widget state:

* column widths;
* scroll position;
* hover state;
* temporary drag state;
* expanded/collapsed tree nodes;
* local visual options.

Global application state must go to `AppState`.

=== Qt Model/View

For lists, tables and trees AVIN GUI should use Qt Model/View architecture.

Preferred Qt classes:

* `QAbstractItemModel`
* `QAbstractTableModel`
* `QTreeView`
* `QTableView`
* `QListView`
* proxy models
* delegates

Examples:

## [source,text]

InstrumentModel -> QTreeView
LevelModel      -> QTableView
TradesModel     -> QTableView
-----------------------------

Qt models are adapters between AVIN data and Qt views.

Qt models must not become global application state owners.

For early vertical slices, simple widgets such as `QTreeWidget` are acceptable, but the long-term direction for non-trivial data is `QAbstractItemModel`.

When storing references in Qt items, prefer stable identifiers such as `Iid`, ids or keys instead of embedding large mutable domain objects directly into GUI items.

=== Graphics scene

Market visualization is based on:

* `QGraphicsView`
* `QGraphicsScene`
* `QGraphicsItem`

Footprint/chart rendering should be organized through scene layers.

Possible layers:

* footprint layer;
* price axis layer;
* time axis layer;
* volume layer;
* delta layer;
* selection layer;
* crosshair layer;
* indicator layers.

Scene items render prepared data.
They do not own market data and do not calculate domain entities.

Preferred data flow:

## [source,text]

Domain data
-> service/analyse calculations
-> ViewModel
-> SceneLayer
-> QGraphicsItem
----------------

Indicators should not become independent mini-applications.
Calculation, state and rendering must be separated.

=== View models

View models may be introduced when raw `AppState` or domain objects become inconvenient for rendering.

View models prepare data for a specific GUI representation.

Examples:

## [source,text]

FootprintViewModel
ClusterViewModel
LevelTableViewModel
InstrumentTreeViewModel
-----------------------

View models are especially useful for scene rendering, tables and inspectors.

They should be introduced when needed, not upfront.

=== Workspace

Workspace architecture is a future extension of `AppState`, not a replacement for `Store`.

Current direction:

## [source,python]

## AppState

Future direction:

## [source,python]

AppState
workspace: WorkspaceState
-------------------------

`WorkspaceState` may later contain:

* opened charts;
* active chart id;
* chart-specific instruments;
* chart-specific timeframes;
* opened dock panels;
* dock layout state;
* loaded datasets;
* selected objects;
* analyse session state.

Workspace should be introduced when the GUI has real workspace needs, for example:

* multiple opened charts;
* active chart switching;
* chart-specific settings;
* saving/restoring layout;
* saving/restoring research sessions.

Do not design a large workspace system before the first working GUI vertical slice.

=== Services and heavy operations

Heavy operations should not live inside widgets.

Examples:

* loading ticks;
* reading parquet files;
* building footprints;
* running analysis;
* receiving live market data.

These operations belong to `data`, `service`, `analyse` or worker components.

GUI dispatches actions and renders results.

For the first GUI stage, AVIN should be offline-first:

## [source,text]

load existing data
build footprint
render footprint
inspect clusters and levels
---------------------------

Broker/live/async integration is not part of the first GUI slice.

=== Async and live data

Qt event loop and asyncio integration must not be solved prematurely.

For future live mode, preferred direction:

## [source,text]

Broker / data worker
-> thread-safe event delivery
-> Store.dispatch(MarketDataArrived)
-> AppState update
-> GUI render
-------------

Live data events should enter GUI through the same action/store/state pipeline.

Widgets must not receive broker events directly.

=== Architecture rule

The central rule:

## [source,text]

GUI widgets do not own application state.
GUI widgets render state and dispatch actions.
Store owns application state.
Domain and service layers remain independent from GUI.
------------------------------------------------------

=== Forbidden patterns

Avoid:

* widget-to-widget business synchronization;
* current instrument stored only inside instrument widget;
* chart widget owning loaded market data as source of truth;
* domain objects inheriting from Qt classes;
* `QGraphicsItem` loading data or building footprints;
* `Store` storing QWidget or QGraphicsItem objects;
* `AppState` knowing about QWidget or QGraphicsItem;
* event bus as replacement for clear state management;
* premature workspace/project/session abstraction.

=== Current accepted direction

AVIN GUI architecture decision:

## [source,text]

Qt-native shell

* MVU/Redux-style application state
* Qt Model/View adapters
* QGraphicsScene render layers
* future WorkspaceState inside AppState

---

This architecture is chosen to keep GUI scalable, debuggable and suitable for a dock-based trading research platform.
